# Hive Resource Structure

Operational knowledge about how Hive resources relate to each other and how to access them on the
collective cluster.

## ClusterDeployment Namespacing

Pool-managed ClusterDeployments live in their **own dedicated namespace**, not in the pool's
namespace (e.g. `app`). The namespace is named after the CD:
- Pool namespace: `app`
- CD namespace: `app-prow-small-aws-420-west2-mxfzb`
- CD name: `app-prow-small-aws-420-west2-mxfzb`

The CD's namespace is set in the ClusterClaim's `status.clusterNamespace` once a claim is
assigned.

## ClusterClaims — Two APIs, Different Access

There are two ClusterClaim APIs on this cluster:

| API group | Kind | Access |
|-----------|------|--------|
| `hive.openshift.io` | ClusterClaim | Per-namespace only (no cluster-scope list) |
| `cluster.open-cluster-management.io` | ClusterClaim | Forbidden for tesshuflower |

Always use the Hive API:
```bash
oc get clusterclaim.hive.openshift.io -n <namespace>
oc get clusterclaim.hive.openshift.io <name> -n <namespace> -o yaml
```

A ClusterClaim connects to its ClusterDeployment via `status.clusterNamespace` (set by Hive when
a cluster is assigned). Before assignment this field is empty and `status.conditions` will show
`Pending / NoClusters` if no ready clusters exist in the pool.

## ClusterPool `spec.size`

`spec.size` is the **standby pool size** — how many pre-provisioned ready clusters Hive keeps warm.
It is NOT a cap on total clusters. A claim against a pool with `size: 0` will still trigger
provisioning; it just means no clusters are kept warm in advance, so the claim will wait for a
fresh install.

## ClusterProvisions

A `ClusterProvision` represents a single install attempt for a ClusterDeployment. Key fields:
- `spec.infraID` — infraID generated by the installer for this attempt (may differ from sibling attempts)
- `spec.stage` — `provisioning`, `complete`, or `failed`
- `spec.attempt` — attempt number (0-based)

**Access**: no cluster-scope list permission. Must query per namespace:
```bash
kubectl get clusterprovision -n <cd-namespace>
```

**InfraID across attempts**: each attempt runs the installer from scratch and may generate a new
infraID. The CD's `spec.clusterMetadata.infraID` is updated to the most recent successful (or
in-progress) attempt's infraID. Previous failed attempts' infraIDs are abandoned — their AWS
resources are genuine orphans.

**Race window**: a `provisioning`-stage ClusterProvision may have an infraID before the CD does.
The `get_live_infra_ids` function only reads from ClusterDeployments, so resources tagged with a
very new infraID could be missed during this brief window. In practice the window is short and
low-resource-count results are unlikely to cause harm, but be aware of it when scanning shortly
after new CDs are created.
